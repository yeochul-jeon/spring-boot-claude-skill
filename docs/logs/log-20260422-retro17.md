# 회고 #17 — spring-security empirical 검증 (탐지 공백 3건 + spring-web 교차 결함 2건)

날짜: 2026-04-22

---

## 사건 요약

`spring-security` SKILL.md 의 구조는 완성 상태였으나(2026-04-21 작성)
실제 코드 생성 후 체크리스트 적용(empirical 검증)은 수행되지 않았다.

서브에이전트로 "세션 기반 로그인 REST API(회원가입/로그인/로그아웃/내 정보 조회)"
시나리오를 `spring-security` SKILL.md 를 참조하지 않은 맨세션 조건에서 생성하고 결함을 수집했다.
5개 결함이 식별되었으며, 모두 기존 체크리스트 항목과 매핑되었다.
신규 원칙 추가는 없고, grep 자동화와 spring-web 참조 누락이 탐지를 막은 구조 공백이었다.

## 발견된 결함

### security-domain 결함 (탐지 공백 3건)

| 결함 | 내용 | 분류 |
|------|------|------|
| S_DP | `new BCryptPasswordEncoder()` 직접 사용 — `DelegatingPasswordEncoder` 아님 | 항목 존재, grep 없음 |
| S4 | `sessionFixation().changeSessionId()` 없음 — 세션 고정 공격 방어 누락 | 항목 존재, grep 없음 |
| S5 | `CorsConfigurationSource` Bean 없음 — CORS 미설정 | 항목 존재, grep 없음 |

세 결함 모두 SKILL.md 체크리스트에 이미 항목이 있었다.
grep 자동 검증 섹션이 없어 수동 확인에만 의존하는 구조였다.

#### S_DP 세부

- 기존 체크리스트: "`DelegatingPasswordEncoder` (BCrypt 기본) 적용, 평문 저장 없음"
- 생성된 코드: `SecurityConfig.passwordEncoder()` 가 `new BCryptPasswordEncoder()` 반환
- `BCryptPasswordEncoder` 는 보안상 문제없으나, `DelegatingPasswordEncoder` 에 비해 알고리즘 마이그레이션 불가
- grep `PasswordEncoderFactories\|DelegatingPasswordEncoder` 로 양성 탐지 가능

#### S4 세부

- 기존 체크리스트: "세션 선택 시: 세션 고정 공격 방어 (`sessionFixation().changeSessionId()`)"
- 생성된 `SecurityConfig`: `sessionManagement(session -> session.sessionCreationPolicy(IF_REQUIRED))` — `sessionFixation` 설정 없음
- 세션 고정 공격(session fixation)은 공격자가 미리 알려진 세션 ID 를 피해자에게 사용하도록 유도하는 취약점이므로, 탐지 공백은 단순 스타일 문제가 아닌 보안 취약점 가능성

#### S5 세부

- 기존 체크리스트: "`CorsConfigurationSource` Bean 등록, `allowedOrigins`가 환경변수·설정값으로 주입"
- 생성된 `SecurityConfig`: CORS 설정 자체가 누락됨 (Spring Boot 기본 CORS 정책 — 브라우저 cross-origin 요청 전부 차단)
- grep `CorsConfigurationSource` 로 양성 탐지 가능

### spring-web 교차 결함 (2건)

| 결함 | 내용 |
|------|------|
| W2 | `@RestControllerAdvice` 없음 — 예외 처리 미구성 |
| W_login | `login()` 에서 `@RequestBody Map<String, String>` 사용 — `LoginRequest` record 아님, `@Valid` 미적용 |

> 두 결함 모두 spring-web SKILL.md 체크리스트가 커버하지만,
> spring-security SKILL.md 는 `spring-principles` 체크리스트 실행만 명시하고
> `spring-web` 체크리스트 실행을 명시하지 않아 탐지 누락.

W_login 는 retro #15 결함 J (Request DTO 미사용)의 반복.
security 시나리오에서 로그인 엔드포인트가 `Map<String, String>` 을 쓰는 것은 자연스러운 흐름이지만
spring-web 원칙상 `@RequestBody` 에는 반드시 `*Request` record 를 사용해야 한다.

## 교훈

1. **체크리스트 항목 존재 ≠ 탐지 가능.** S_DP·S4·S5 모두 항목이 있었으나
   grep 패턴 없이 수동 확인에만 의존했다. grep 섹션 추가로 이후 세션은 커맨드 한 줄로 탐지 가능.

2. **보안 결함은 "스타일 위반"이 아닌 "취약점"으로 분류해야 한다.**
   S4(세션 고정 방어 누락)는 실제 공격 경로로 이어지는 취약점이다.
   SKILL 이 이 항목을 단순 체크박스로만 제시했다면 맥락을 잃기 쉽다.
   `session-auth.md` 에 공격 시나리오와 함께 설명이 있는 이유가 여기에 있다.

3. **security 시나리오에서도 spring-web 결함 2건이 발생했다.**
   login 엔드포인트에서 `Map<String, String>` 은 자연스럽게 나타나는 패턴이므로,
   spring-security SKILL 이 `spring-web` 체크리스트 실행을 명시해야
   교차 결함을 체계적으로 탐지할 수 있다.
