## 발견된 결함

### B1_GREP — chunk() type witness false-negative

- v0.1 grep 패턴: `\.chunk(`
- 실제 생성 코드: `.<Member, MemberExportDto>chunk(CHUNK_SIZE, transactionManager)`
- `>chunk(` 이므로 `\.chunk(` 패턴에 매치되지 않아 WARN 반환 (false-negative)
- 분류: **탐지 공백** — chunk size 는 실제로 설정됨(CHUNK_SIZE=1000)
- 수정: `chunk(` 로 변경 + `//\|chunkSize\|CHUNK_SIZE\s*=` filter 추가

### B3_DS — JobRepository DataSource 분리 탐지 공백

- BASELINE 코드: `spring.datasource` 단일 설정, `@BatchDataSource` Bean 없음
- v0.1 B3 grep: `JobRepository\|BatchDataSource\|batchDataSource` → `JobRepository` import 에 매치 → false-PASS
- 분류: **탐지 공백** + **실제 절대 원칙 위반** (원칙 4: DataSource 분리 필수)
- 수정:
  - grep 패턴에 `EnableJdbcJobRepository` 추가 (Spring Boot 4 패턴)
  - `JobRepository` 단독 매치를 PASS로 인정하지 않는 주석 추가
  - references/job-design.md §5 에 Before 안티패턴 블록 추가

## Spring Boot 4 API 변경 — B3 관련 주의

2차 회전에서 subagent 가 `@BatchDataSource` 대신 `@EnableJdbcJobRepository(dataSourceRef=...)` 를
사용했다. Spring Batch 6 (Boot 4) 에서의 DataSource 분리 공식 방법이다.

- Boot 3.x: `@BatchDataSource` Bean 또는 `DefaultBatchConfiguration` 서브클래스
- Boot 4.x: `@EnableJdbcJobRepository(dataSourceRef="batchDataSource", transactionManagerRef="batchTransactionManager")`

SKILL.md grep B3 은 두 패턴을 모두 탐지하도록 업데이트됐다.

## 교훈

1. **Java generic type witness 는 grep의 맹점이다.**
   `.<T, R>method()` 구문에서 `.` 와 `method` 사이에 `<...>` 가 끼면
   `\.method(` 패턴이 작동하지 않는다. Spring Batch `chunk()` 외에도
   유사한 builder 메서드(`.<T>read()`, `.<T>write()`)에서 동일 문제가 발생할 수 있다.
   앞으로 grep 패턴은 선행 `.` 를 생략하거나 `[.>]method(` 로 작성한다.

2. **"1건 이상이면 PASS" 패턴은 탐지 대상이 구체적이어야 한다.**
   B3 에서 `JobRepository` import 를 탐지해 PASS 처리한 것은
   "DataSource 분리 존재 여부"가 아닌 "JobRepository 사용 여부" 를 탐지한 셈이다.
   grep 대상을 **실제 분리 설정을 나타내는 심볼**(`@BatchDataSource`,
   `EnableJdbcJobRepository`)로 좁혀야 false-PASS를 막을 수 있다.

3. **Spring Boot 버전이 올라갈수록 grep 패턴 유지보수가 필요하다.**
   Boot 3 → Boot 4 전환으로 B3 탐지 심볼이 바뀌었다(`@BatchDataSource` →
   `@EnableJdbcJobRepository`). 스킬 grep 패턴은 현재 사용 중인 Boot 버전
   (fetch-latest-versions.sh 반환값) 에 맞게 정기 점검이 필요하다.
