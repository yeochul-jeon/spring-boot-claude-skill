## 발견된 결함

### AS5_EXCEPTION — CompletableFuture 예외 처리 탐지 없음

- v0.1 에 `exceptionally`/`handle` 검사 grep 없음
- `@Async` 메서드 반환값을 호출 측에서 받아 예외 처리 누락 시 완전 무시됨 — 미탐지
- 분류: **탐지 공백** — BASELINE 코드는 `exceptionally` 정상 사용(latent)
- 수정: AS5 신규 grep 추가 (`grep -rn "exceptionally\|\.handle(" <SRC>/`)

### AS1_DIRECT_THREAD — 원시 스레드 생성 미탐지

- v0.1 AS1: `SimpleAsyncTaskExecutor` 만 탐지
- `new Thread(` 직접 생성 / `Executors.newCachedThreadPool()` (무한 스레드풀) 미탐지
- 분류: **탐지 공백** (latent)
- 수정: AS1 확장 패턴 — `UNSAFE` 변수 기법으로 캡처

### AS1_HEAD_BUG — head -3 && echo "WARN" 패턴 버그

- `grep ... | head -3 && echo "WARN"` — `head`는 빈 입력에서도 exit 0 반환
- `&&` 조건이 항상 참 → 빈 결과에도 "WARN" 항상 출력 = false-positive
- 분류: **구현 버그** (grep 패턴 설계 결함)
- 수정: `UNSAFE=$(grep ...)` / `[ -n "$UNSAFE" ] && ... || echo "PASS"` 패턴으로 교체

### AS3_MULTILINE — 어노테이션 끼임 false-positive

- v0.1 AS3: `grep -A1 "@Async"` — 바로 다음 줄만 검사
- `@Async` 뒤에 `@Transactional` 등 다른 어노테이션이 오면 해당 어노테이션이 반환 타입으로 오인됨
- 분류: **탐지 공백** (false-positive, latent)
- 수정: MANUAL 수동 확인 격상 ("반환 타입인지 중간 어노테이션인지 확인")

## 교훈

1. **grep 구현에도 버그가 있다 — 테스트하고 검증해야 한다.**
   `head -3 && echo "WARN"` 패턴은 `head` 의 종료 코드 특성 때문에 항상 WARN 을 출력한다.
   grep 패턴은 실제 sandbox 에 적용해서 false-positive/false-negative 를 확인해야 한다.

2. **caller-side 예외 처리는 별도 grep 항목이 필요하다.**
   `@Async` 메서드의 반환 타입(AS3)과 호출 측 예외 처리(AS5)는 다른 파일에 있다.
   한 번의 grep으로 커버할 수 없으므로 체크 항목을 분리해야 한다.

3. **`-A1` 멀티라인 grep 은 어노테이션 끼임에 취약하다.**
   `grep -A1 "@Annotation"` 로 다음 줄을 검사하는 방식은 중간에 어노테이션이
   끼어 있으면 실제 타겟 라인을 놓친다. W5a 방식(`tr '\n' ' '` 후 regex)이 더 robust 하지만
   복잡도가 높아 이번에는 MANUAL 확인으로 격상.
