# 회고 #21 — W5a multi-line grep false-positive 해소

날짜: 2026-04-22

---

## 사건 요약

retro #20 직후 이전 세션 브리핑에 "W5a 는 retro #15~#17 에서 누적 deferred 된
미해결 항목" 으로 기재돼 있었다. 본 사이클에서 실제 retro 로그를 재검증한 결과,
**W5a 가 공식 deferred 항목으로 기록된 적이 없음**을 확인했다.

다만 실제 grep 패턴(`grep -rn "@RequestBody" ... | grep -v "@Valid"`) 을
검토하니 `@RequestBody` 와 `@Valid` 가 별개 라인에 선언된 경우 (IDE 포맷터가
긴 파라미터를 줄바꿈 분리할 때 자연 발생) **false-positive 노이즈**를 내는 결함이
존재했다. 실측 실패 0건(latent)이지만 노이즈로 인한 "PASS 로 퉁치기" 2차 위험
경로를 차단하기 위해 본 사이클에서 해소한다.

### 스캐너 설계 상세

```bash
for f in $(find $SRC -path '*/controller/*.java' 2>/dev/null); do
  HITS=$(tr '\n' ' ' < "$f" \
    | grep -oE '@RequestBody[^,)]{0,200}[,)]' \
    | grep -v '@Valid')
  if [ -n "$HITS" ]; then
    printf '%s:\n%s\n' "$f" "$HITS"
    W5A_MISS="yes"
  fi
done
[ -z "$W5A_MISS" ] && echo "PASS"
```

설계 포인트:

1. **파일 단위 처리** — `tr '\n' ' '` 로 개행 정규화 후 grep. 파일 경계를
   넘는 오탐 방지.
2. **파라미터 경계 종결** — `[^,)]{0,200}` 로 첫 `,` 또는 `)` 까지만 캡처.
   상한 200자는 정상 파라미터 길이 초과 방어 (runaway match 방지).
3. **line number 손실 허용** — 대신 파일 경로 + 오프딩 조각 출력.
   정확한 라인은 IDE/에디터로 `@RequestBody` 검색 시 즉시 확인 가능.
4. **generics 안전성** — `Map<K,V>` 내부 `,` 는 W5b(`@RequestBody Map<` 금지)
   로 별도 차단되므로 본 스캐너의 generics 내 `,` 조기 종결은 문제 없음.

### 검증 (실측)

`/tmp/w5a-test/` 에서 4 케이스 fixture 로 스캐너 동작 확인:

| 케이스 | 선언 | 기대 | 결과 |
|--------|------|------|------|
| m1 | `@RequestBody @Valid FooReq req` (single) | miss 없음 | ✅ PASS |
| m2 | `@RequestBody\n@Valid FooReq req` (multi) | miss 없음 | ✅ PASS — **핵심 개선** |
| m3 | `@RequestBody FooReq req` (single, 누락) | miss 히트 | ✅ FAIL 탐지 |
| m4 | `@RequestBody\nFooReq req` (multi, 누락) | miss 히트 | ✅ FAIL 탐지 |

retro #20 당시 line-oriented grep 이라면 m2 가 false-positive 였음.

## 교훈

1. **브리핑의 "deferred 기록" 을 로그로 교차 확인하라.**
   이전 세션 요약에 "retro #15~#17 누적 deferred" 로 기재됐으나 실제 retro
   로그에는 흔적 없음. 세션 간 상태 전이 시 **요약 vs 1차 기록** 불일치를
   기본 의심해야 한다.

2. **latent 결함도 `tool 철학 보존` 선에서는 고치는 게 싸다.**
   실측 실패 0건이었지만 수정 비용이 POSIX grep 한 블록 교체로 끝나는 이상,
   미래 사용자 코드(IDE 포맷팅 적용) 에서 터질 위험을 선제 차단하는 편이
   총비용 낮다.

3. **codex 리뷰 도구는 diff 기반이다.**
   `/codex:review`·`/codex:adversarial-review` 두 도구 모두 branch diff 가
   없으면 리서치 분석이 불가. 본 사이클에서 리서치 단계에 두 도구를 호출했으나
   둘 다 "approve, no material findings" 로 반환되어 사실상 no-op 였다.
   리서치 단계는 Explore 서브에이전트 + 수동 분석으로 처리하고 codex 는
   커밋 diff 가 있는 단계에서만 호출하는 것이 효율적.
