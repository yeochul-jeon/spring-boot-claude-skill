## 패치 상세

### 1. A7d — whitespace 허용 (`chunk\s*\(`)

**문제**: `chunk(` 리터럴은 `.chunk (...)` whitespace 변형 미탐지.
**수정**: `-E` + `chunk\s*\(` — 공백 0+ 허용.

### 2. A8b / A10d — FQCN 대체

**문제**: `@EnableCaching` / `@EnableAsync` 는 FQCN 형식 미탐지.
```java
@org.springframework.cache.annotation.EnableCaching
@org.springframework.scheduling.annotation.EnableAsync
```
**수정**: alternation 추가 — `@EnableCaching|springframework\.cache\.annotation\.EnableCaching`.

### 3. A8d — `@TimeToLive` / `time-to-live` 추가

**문제**: Redis Repository `@TimeToLive`, properties `spring.cache.redis.time-to-live` 미탐지.
**수정**: alternation 추가.

### 4. A10e — `handleAsync` 추가

**문제**: `future.handleAsync(...)` 는 `\.handle(` 에 안 걸림.
**수정**: `|handleAsync` alternation 추가.

### 5. A7b / A7c / A8c / A9c — known-limitation 주석

이들은 존재-확인 규칙의 구조적 한계:
- A7b/A7c: 목차·다이어그램 라인만으로도 PASS
- A8c: `disableCachingNullValues()` 단독 null 방어만 있고 `unless` 없는 경우 FAIL 가능
- A9c: `"LogstashEncoder 사용 금지"` 같은 부정 문장도 통과

이 구조는 grep만으로 fix 불가 — 실질 위반은 BASELINE empirical 사이클이 커버함.
SKILL.md 본문에 해당 제약을 주석으로 명시해 유지보수자가 혼동하지 않게 함.

## 교훈

1. **alternation 확장은 존재-확인 규칙의 FN 대부분을 해결한다.**
   Java FQCN(`org.springframework...`), Kotlin alias, 메서드 variant(`handleAsync`) 모두
   `-E "A|B|C"` 한 줄로 커버됨. 체감 복잡도 대비 탐지율 향상이 크다.

2. **grep-based 정적 검사는 문맥(긍정/부정/anti-pattern) 판별을 못한다.**
   `"사용 금지"` 부정 문장을 "존재함" 으로 오인하는 문제(A9c)는 자연어 분석 없이 해결 불가.
   구조적 한계를 인정하고 empirical 사이클로 보완하는 것이 맞다.

3. **known-limitation 을 명시적으로 주석에 남겨야 한다.**
   "이 grep 은 왜 단순한가" 를 나중에 보는 사람이 이해할 수 있도록 SKILL.md/regression 주석에
   `known-limitation` 으로 표식. 향후 재리뷰 시 같은 지적이 반복되는 것을 방지.
