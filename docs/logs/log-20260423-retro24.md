## 교훈

1. **잘 알려진 패턴은 BASELINE PASS 가능하다.**
   spring-cache의 @Cacheable/TTL/null 방지 패턴은 업계 표준이 명확해
   SKILL 없이도 모범 구현이 나올 수 있다. 반대로 spring-batch DataSource 분리처럼
   프레임워크 내부 메커니즘(JobRepository 격리)은 BASELINE 실패 가능성이 높다.

2. **grep 패턴은 TTL 설정 위치(Java vs yml)를 모두 커버해야 한다.**
   `entryTtl` 은 Java config, `time-to-live` 는 yml 설정 — 두 경로가 동등한 효과를
   내므로 grep도 두 경로를 모두 스캔해야 false-PASS를 막을 수 있다.

3. **TTI(Time-To-Idle)도 TTL 요건 충족 수단이다.**
   Caffeine의 `expireAfterAccess`는 마지막 접근 후 만료되는 TTI 설정으로,
   자주 읽히는 데이터에는 오히려 TTL보다 적합할 수 있다. grep 패턴이
   TTL 의미의 API만 검사하면 TTI 설정을 "TTL 없음"으로 오탐한다.
