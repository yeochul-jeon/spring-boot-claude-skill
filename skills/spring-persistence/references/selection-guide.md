# JPA vs MyBatis 선택 가이드

`spring-persistence` 진입 시 **가장 먼저 참조하는 문서**.  
프레임워크를 가정하지 말고, 이 체크리스트로 추천 → 사용자 승인 순서로 진행한다.

---

## 결정 체크리스트

아래 7개 항목을 평가해서 JPA·MyBatis 가중치를 합산한다.

| # | 평가 항목 | JPA 적합 | MyBatis 적합 |
|---|---|---|---|
| 1 | **도메인 복잡도** | 비즈니스 규칙이 많고 객체 그래프가 복잡 | 단순 CRUD, 데이터 위주 |
| 2 | **쿼리 복잡도** | 단순 조회·조건 필터 위주 | 복잡한 통계·조인·서브쿼리 다수 |
| 3 | **쿼리 튜닝 요구** | ORM 추상화 수용 가능 | 실행 SQL을 직접 제어해야 함 |
| 4 | **동적 쿼리 비율** | 낮음 (조건 변수가 적음) | 높음 (검색 필터, 보고서 쿼리) |
| 5 | **레거시 SQL/스키마** | 새 스키마 설계 가능 | 기존 테이블·뷰·프로시저 연동 필요 |
| 6 | **팀 JPA 숙련도** | 팀이 JPA에 익숙 | 팀이 SQL에 더 익숙 |
| 7 | **테스트 용이성** | 객체 단위 테스트 + `@DataJpaTest` | SQL 정확성 검증이 핵심 |

### 채점 방식

각 항목에서 해당하는 쪽에 1점. 총 7점.

- **JPA 4점 이상**: JPA 추천
- **MyBatis 4점 이상**: MyBatis 추천
- **동점(3.5) 또는 혼합**: 하이브리드 케이스 검토 (아래 참조)

---

## 하이브리드 사용 케이스

한 프로젝트에서 둘 다 쓰는 것은 **예외적 케이스**에만 허용한다.

| 시나리오 | 권장 |
|---|---|
| 대부분 도메인 조작 + 일부 통계 조회 | JPA 기본, 통계 쿼리만 Native Query 또는 `JdbcTemplate` |
| 대부분 보고서 쿼리 + 일부 엔티티 저장 | MyBatis 기본, 저장 로직만 JPA |
| 레거시 테이블 + 신규 도메인 혼재 | MyBatis(레거시) + JPA(신규) 모듈 분리 |

하이브리드를 선택하면 트랜잭션 경계와 DataSource 설정이 복잡해진다 —  
`transaction.md`의 전파 속성 섹션을 반드시 확인한다.

## 사용자에게 제시하는 템플릿

체크리스트를 채점한 뒤 아래 형식으로 결과를 제시하고 확인받는다.  
(참고: `spring-init/references/decision-tree.md` 제시 스타일과 동일)

```
영속성 프레임워크 분석:
- 도메인 복잡도: {평가} → JPA/MyBatis +1
- 쿼리 복잡도: {평가} → JPA/MyBatis +1
- 쿼리 튜닝 요구: {평가} → JPA/MyBatis +1
- 동적 쿼리 비율: {평가} → JPA/MyBatis +1
- 레거시 SQL/스키마: {평가} → JPA/MyBatis +1
- 팀 숙련도: {평가} → JPA/MyBatis +1
- 테스트 용이성: {평가} → JPA/MyBatis +1

점수: JPA {J} / MyBatis {M}
추천: **{선택}** (근거: {한 줄 이유})

이 분석이 맞나요? 다른 선택을 원하시면 지정해 주세요.
```

사용자가 점수와 다른 선택을 하면 이유를 한 번 확인하고, 대답 들으면 그대로 진행한다 (잔소리 금지).

---

## 원칙

- 선택은 **추천**이지 강제가 아니다. 사용자 최종 승인이 항상 우선한다.
- 팀 숙련도는 과소평가하지 않는다 — 익숙한 도구가 이론적 최선보다 낫다.
- 결정이 나면 해당 references 파일을 즉시 참조:
  - JPA → `jpa.md`
  - MyBatis → `mybatis.md`
  - 트랜잭션 경계는 선택과 무관하게 `transaction.md` 참조
