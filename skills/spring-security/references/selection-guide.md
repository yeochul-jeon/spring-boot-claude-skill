# 세션 vs JWT 선택 가이드

`spring-security` 진입 시 **가장 먼저 참조하는 문서**.
프레임워크를 가정하지 말고, 이 체크리스트로 추천 → 사용자 승인 순서로 진행한다.

---

## 결정 체크리스트

아래 6개 항목을 평가해서 세션·JWT 가중치를 합산한다.

| # | 평가 항목 | 세션 적합 | JWT 적합 |
|---|---|---|---|
| 1 | **클라이언트 유형** | 서버 렌더링(MVC), 전통적 웹 브라우저 | SPA (React/Vue), 모바일 앱 |
| 2 | **서버 구성** | 단일 서버 또는 고정 세션 저장소(Redis) 수용 가능 | 수평 확장, 무상태(stateless) 필요 |
| 3 | **토큰 즉시 무효화 요구** | 로그아웃·권한 변경 시 즉시 차단 필요 | 만료까지 기다릴 수 있거나 블랙리스트 허용 |
| 4 | **외부 API / 마이크로서비스** | 내부 서비스 간 호출 없음 | 서비스 간 토큰 전달, API Gateway 인증 |
| 5 | **보안 민감도** | 금융·의료 등 세션 제어 강도 요구 | 일반 서비스, 토큰 만료 주기로 위험 관리 |
| 6 | **팀 경험** | Spring MVC + 세션 경험 있음 | REST API + JWT 구현 경험 있음 |

### 채점 방식

각 항목에서 해당하는 쪽에 1점. 총 6점.

- **세션 4점 이상**: 세션 기반 추천
- **JWT 4점 이상**: JWT 추천
- **동점(3)**: 서비스 특성 재확인 후 결정. 일반적으로 REST API 신규 프로젝트라면 JWT 권장

---

## 하이브리드 케이스

세션과 JWT를 동시에 운영하는 것은 **예외적 케이스**에만 허용한다.

| 시나리오 | 권장 |
|---|---|
| 웹 관리자 페이지 + 모바일 API 공존 | 관리자: 세션, API: JWT. `SecurityFilterChain` 분리 |
| OAuth2 소셜 로그인 추가 | JWT 기반에 `spring-security-oauth2-client` 추가 |
| 레거시 세션 서비스 + 신규 API | 레거시 그대로, 신규 엔드포인트만 JWT 필터 적용 |

## 사용자에게 제시하는 템플릿

체크리스트를 채점한 뒤 아래 형식으로 결과를 제시하고 확인받는다.

```
인증 방식 분석:
- 클라이언트 유형: {평가} → 세션/JWT +1
- 서버 구성: {평가} → 세션/JWT +1
- 토큰 무효화 요구: {평가} → 세션/JWT +1
- 외부 API: {평가} → 세션/JWT +1
- 보안 민감도: {평가} → 세션/JWT +1
- 팀 경험: {평가} → 세션/JWT +1

점수: 세션 {S} / JWT {J}
추천: **{선택}** (근거: {한 줄 이유})

이 분석이 맞나요? 다른 선택을 원하시면 지정해 주세요.
```

사용자가 점수와 다른 선택을 하면 이유를 한 번 확인하고, 대답 들으면 그대로 진행한다 (잔소리 금지).

## 원칙

- 선택은 **추천**이지 강제가 아니다. 사용자 최종 승인이 항상 우선한다.
- 결정이 나면 해당 references 파일을 즉시 참조:
  - 세션 → `session-auth.md`
  - JWT → `jwt-auth.md`
  - 비밀번호 인코더는 선택과 무관하게 `password-encoding.md` 참조
  - CORS는 선택과 무관하게 `cors.md` 참조
